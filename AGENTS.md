## Learned User Preferences

- When auditing or changing data paths, cross-check table and column names against `prisma/schema.prisma` and `prisma/migrations`, not assumptions from older SQL scripts.
- For security and tenancy reviews, prefer concrete file references, severities, and apply-ready fixes over high-level advice alone.
- For hub, storefront, POS, public-store, and marketing mobile work: keep desktop layouts unchanged; use dual-layout with `lg:` breakpoints (`hidden lg:block` / `lg:hidden`) — compact app-style tile hubs on small screens for hub; marketing pages use shared tokens in **`lib/utils/marketingLayout.js`** and **`MarketingSection`** (`components/marketing/layout/MarketingSection.jsx`).

## Learned Workspace Facts

- **Next.js App Router**: files with top-level **`'use server'`** export only **async** Server Actions (move sync helpers to **`lib/utils/*`**). Root **`proxy.ts`** exports **`proxy()`** (not middleware) for security headers/metadata only—tenant/business resolution stays in Node route handlers (Prisma/pg), not in the proxy file.
- **RSC / server actions → Client Components**: Prisma **`Decimal`** values are not serializable across the boundary (e.g. **`customers.credit_limit`**, **`outstanding_balance`**, **`opening_balance`**). Run payloads through **`serializeDecimalsDeep`** from **`lib/utils/serializePrismaDecimals.js`** before returning them to the client; do not rely on **`instanceof Prisma.Decimal`** alone when duplicate **`@prisma/client`** bundles can break type checks—use the helper’s decimal-like detection.
- Tenant-style isolation in this app is `business_id` scoped to `businesses`; there is no separate `tenant_id` column family across the main schema. **Forms and DB writes**: Prisma `findUnique` on tenant tables is **not** tenant-filtered by the extension—use `findFirst` with `business_id` and/or `assertEntityBelongsToBusiness` before mutations; multi-step writes should use transactions. See **`docs/DATA_INTEGRITY_AND_FORMS.md`**.
- Prisma migrations under `prisma/migrations` are the canonical database evolution path; `scripts/migrations` and `lib/db/migrations` are historical or manual—treat them as non-authoritative unless explicitly reconciled. Legacy `invoice_payments` rows may omit columns the app expects (`payment_method`, `received_by`, earlier `business_id` / soft-delete fields). Prefer Prisma **`20260606_invoice_payments_record_payment_columns`** and **`20260607_invoice_payments_received_by`**, or the idempotent SQL under `lib/db/migrations/038_invoice_payments_record_payment_columns.sql` and `039_invoice_payments_received_by.sql`, per **`docs/DATABASE_MIGRATIONS.md`**. Prisma maps `invoice_payments.received_by` / `deleted_by` as plain strings so values stay compatible with **`user`.id** (text).
- Customer AR balance truth uses **`invoice_payments`** and **`calculate_invoice_balance()`** (`InvoicePaymentService`, `PaymentModal`, API v1 invoice payments). Legacy **`payments`** + **`payment_allocations`** (`PaymentService`, `PaymentManager`) is a separate ledger path and does not reduce invoice balance unless explicitly routed through **`invoice_payments`**. Canonical payment types: **`lib/utils/paymentTypes.js`** (`receipt`/`payment`). **Purchase PO status** canon: **`lib/constants/purchaseStatus.js`** (`draft`/`ordered`/`received`/`cancelled`; **`normalizePurchaseStatus`** maps `sent`/`pending` → `ordered`) — use in Zod schemas, API routes, and PO/GRN UI.
- Hub inventory display stock comes from **`ProductService.resolveDisplayStock()`** (sum **`product_stock_locations`**, then batch/reserved qty, then variants/headline **`products.stock`**). Canonical **writes** use **`InventoryService`** (`product_stock_locations` → **`syncProductStock`**); **`batchAllocation.js`** / **`reservationManagement.js`** are unwired — FIFO/reservations live in **`InventoryService`**. Public storefront checkout (**`POST /api/storefront/[domain]/orders`**) decrements headline/variant stock via direct SQL and **bypasses** InventoryService (locations, movements, reservations). **`upsertIntegratedProductAction`** in **`lib/actions/premium/automation/inventory_composite.js`** must run stock add/remove on the **same pg client** (`invokeAddStockInTx` / `invokeRemoveStockInTx`); nested **`addStockAction`** opens another connection and can throw **Product not found** on uncommitted inserts. Placeholder serial rows must not force batch/serial mode; explicit headline **`stock`** is reconciled after batch/serial handling so inline grid edits persist.
- **Multi-country onboarding**: use **`lib/utils/regionalHelpers.ts`** (`getRegionalStandards`, `getRegistrationCountryOptions`) and **`docs/REGIONAL_STANDARDS.md`**. Registration stores `businesses.country` as the **canonical country name**; wizard/API may send **ISO or legacy aliases**. Extend **`REGIONAL_REGISTRY`** + **`ALIAS_TO_ISO`** before adding currencies in **`lib/currency/index.ts`**. Country-aware brand catalogs, payment gateways, and tax overlays resolve via **`lib/regionalMarket/`** (`getBrandsForMarket`, `getMarketFeatures`, **`domainBrandMap.js`**); pass **`countryIso`** into **`getDomainKnowledge`** and run **`bun run verify:regional-market`** after catalog edits.
- **Verticals (62 keys)**: canonical presets live in **`lib/domainData/*.js`** → **`lib/domainKnowledge.js`**; each row should define **`intelligence`** (seasonality, peaks, perishability, lead time, etc.) for Industry Insights, Smart Restock, and AI forecasting. **`getDomainKnowledge`** merges defaults and supplies a **`name`** from the slug when the row omits it. Run **`bun run verify:domains`** after edits; see **`docs/DOMAIN_VERTICALS.md`**.
- **Market readiness / form wiring**: Hub forms should use **`lib/utils/formErrorHandler.jsx`** with **`actionSuccess` / `actionFailure`** from server actions; launch QA and scope are summarized in **`docs/MARKET_READINESS.md`** (not a substitute for end-to-end QA). Run **`bun run verify:mvp-launch`** after MVP launch wiring edits (OTP prod fail-loud, Stripe webhook **503** when unconfigured, Easy Mode **finance**/**gst** nav, auth confirmed onboarding redirect). Default SaaS trial is **14 days** via **`TRIAL_CONFIG`** in **`lib/config/platform.js`** (`getTrialExpiry`, Stripe checkout trial). Admin platform feature flags use `platform_feature_flags` and `platform_feature_flag_overrides`. The Prisma `feature_flags` model (`business_id`, `feature_name`) exists but is **not** wired into hub plan guards today — **per-tenant module overrides** use `businesses.settings.packaging` (`planHasFeatureWithPackaging`). Business **owners** edit packaging from **Settings → Billing** via **`updateOwnerBusinessPackagingAction`** (`settings.packaging` permission); platform staff use **`updateBusinessPackaging`** in `lib/actions/admin/platform.js`.
- **Hub mobile (below `lg`)**: tile hubs (`InventoryMobileHub`, `DashboardMobileHub`, `StorefrontMobileHub` + **`StorefrontTabShell`**); shared **`components/mobile/MobileHubPrimitives.jsx`**, form tokens **`lib/utils/formMobileStyles.js`**, FAB offset **`lib/utils/mobileLayout.js`** (`MOBILE_BOTTOM_NAV_CLASS`); **`HubMobileBottomNav`** + **`useHubMobileNav`**; child managers use **`useStorefrontEmbedded()`** inside the shell to skip duplicate headers — see **`docs/STOREFRONT_MOBILE_NAV.md`**. Public customer storefront mobile uses **`StoreMobileBottomNav`** in **`app/store/[businessDomain]/layout.jsx`** (deduped homepage sections, no duplicate headers/banners).
- Stripe webhooks and other cross-tenant billing writes should use `prismaBase` from `lib/db.js`, not the tenant-extended `db` client, especially for models like `stripe_webhook_events`. **Public storefront** uses raw `pool` SQL (not Prisma tenant extension): checkout **`POST /api/storefront/[domain]/orders`** ( **`createStorefrontOrder`** is deprecated), **`resolveStorefrontBusiness`**, **`requireStorefrontHubAccess`** for hub actions; stock/cart require **`businessId`**; run **`npm run verify:storefront-tenancy`** after storefront edits. Online sales fulfill via hub **orders** tab → **`OrdersManager`** / **`storefront_orders`** (not **`invoices`**). Commerce ledgers are separate: **`storefront_orders`**, **`pos_transactions`**, **`invoices`**, **`restaurant_orders`**. Finance flows that post journals should use `journal_entries` and `gl_entries` shapes from the schema (e.g. via `AccountingService`), not legacy column names from abandoned SQL sketches. **`POS`** and other raw-SQL services must match Prisma column names for `pos_transaction_items` (e.g. `total_amount`, required `product_id`) to avoid runtime drift; domain document labels use **`getDomainConfig(category).label_overrides`** (not `.labels`) in POS/print/PDF.
- `lib/db.js` documents TLS behavior for the pool via `.env.example` (`DATABASE_SSL_INSECURE`, `DATABASE_SSL_CA_PATH`, `DATABASE_SSL_DISABLE`, hosted-host heuristics); pool vs `prismaBase` vs tenant `db` tradeoffs and related risks are summarized in **`docs/AUDIT_SCHEMA_AND_INTEGRATIONS.md`**. Platform owner detection is driven by env allowlists (`PLATFORM_OWNER_EMAILS`, plus legacy merge vars). Business hub **Pending Orders** in `RemindersPortlet` counts **invoice** pending/processing (not **`storefront_orders`**) and must navigate to the **`invoices`** tab (not **`purchases`**); map **`view-*`** quick actions via **`QUICK_VIEW_ACTION_TO_TAB`**. Alerts are split: in-app **`notifications`** + **`NotificationBell`**/SSE, inventory **`low_stock_alerts`** + dashboard KPIs (`LowStockAlerts`, `RemindersPortlet`), and Resend email for customer/order/contact — not a single unified pipeline.
